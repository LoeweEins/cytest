# cytest 电商平台 E2E 测试计划（最详细版）

```bash
cd /path/to/saleor-platform
docker compose up
```

> 适用框架：`z_cytest-main`（cytest）  
> 测试类型：系统测试 / 端到端（E2E）自动化为主，配合必要的手工探索测试  
> 输出物：Vue 报告（`log/vue_report_*.html`）+ 文本日志（`log/testresult.log`）

## 1. 参考的开源电商项目（远程 API，可直接做自动化）

为满足“**不需要本地部署**、可以直接写 API 自动化”的目标，本计划选择一个**开源且提供公开可访问 Demo API** 的电商项目作为业务蓝本与接口来源。

### 1.1 主参考项目（推荐：Saleor）

- **项目**：`saleor/saleor`（开源电商平台，GraphQL API）
- **开源仓库**：`https://github.com/saleor/saleor`
- **公开 Demo（Storefront + API）**：
  - Storefront（人可视化验证数据）：`https://demo.saleor.io/graphql/`
  - GraphQL API Endpoint（自动化直接调用）：`https://demo.saleor.io/graphql/`
- **官方 API 文档**：`https://docs.saleor.io/`

> 说明：本计划以“电商通用业务能力 + Saleor GraphQL 的公开 Demo API”为落地对象。你可以在不部署任何后端的前提下，直接用 cytest 编写 API 自动化，并生成可读性很强的 Vue 报告。

## 2. 目标与范围

### 2.1 测试目标（Why）

- **业务目标**：验证电商平台核心交易链路（浏览 → 加购 → 下单 → 支付/取消 → 发货/收货 → 退款）可用、稳定、可回归。
- **质量目标**：在每次迭代（或每日构建）中，用自动化回归覆盖高风险路径，尽早发现交易链路断裂、权限漏洞、库存一致性问题、价格计算错误等严重缺陷。
- **工程目标**：用 cytest 的“目录=套件、Signal 解耦、Vue 报告、检查点左右值展示”能力，把系统测试的可读性与可维护性做出来（适合测试/产品/业务共同查看）。

### 2.2 测试范围（What）

- **覆盖的业务域**（用户侧）：
  - 用户与鉴权：注册/登录/登出、Token 续期、权限与风控（验证码/限流）
  - 商品域：类目、搜索、商品详情、SKU/规格、价格、上下架、库存展示
  - 购物车：加购、修改数量、删除、勾选、合并购物车（登录前后）
  - 订单域：确认订单、地址、运费、优惠券、下单、取消、支付结果同步
  - 支付域：支付发起、回调/对账、异常支付、重复回调幂等
  - 履约域：发货、物流、签收、售后（退货退款/仅退款）、评价
  - 营销域：优惠券、满减、秒杀/限购（如项目具备）
- **覆盖的业务域**（运营/管理侧）：
  - 商品管理：上新、改价、库存调整、上下架
  - 订单管理：发货、退款审核、异常单处理
  - 营销管理：券模板、活动开关、库存预热
  - 权限管理：角色/菜单、审计日志（如项目具备）

### 2.3 不在范围（What not）

- 非功能性：大规模压测（可另立性能专项）；安全攻防（可另立安全专项）
- 多端（iOS/Android）专项兼容（本计划以 Web/H5 为主）
- 财务/对账的全链路自动化（可先做关键接口校验）

## 3. 测试环境与数据策略（How）

### 3.1 环境分层（远程 Demo 约束版）

当我们使用公开 Demo API 时，环境分层会简化为：

- **DEMO**：公开 Demo 环境（不可控、可能随时重置数据、可能有限流）
- （可选）**自建 SIT/UAT**：当你需要写入类操作（创建商品/改价/发货）或稳定回归时，再切换到自建环境

### 3.2 数据策略（强烈建议：面向 Demo 环境）

系统测试最大的难点是“有状态”。为保证可回归与可重复执行：

- **数据准备原则**：
  - Demo 环境中优先选择“只读/低副作用”的链路（浏览/搜索/商品详情/类目/集合）
  - 需要写入（如下单/创建 checkout）时，尽量使用随机化输入、幂等键，并允许数据被系统定期重置
- **关键主数据**：
  - 用户：普通用户、黑名单用户、VIP 用户
  - 地址：默认地址、无效地址（缺省字段）
  - 商品：普通商品、SKU 多规格商品、库存紧张商品、下架商品
  - 营销：可用券、过期券、门槛券（满减）
- **幂等与隔离**：
  - 订单创建需保证唯一（用时间戳/随机串）
  - 支付回调要能重复触发而不产生重复入账（验证幂等）

## 4. 测试方法与分层（API + UI）

### 4.1 推荐组合（以 API 为主）

- **API 为主、UI 为辅**：
  - 核心交易链路：API 自动化覆盖 70%（更稳定、更快）
  - 关键 UI 路径：UI 自动化覆盖 20%（验证真实交互与前端集成）
  - 探索性手工：10%（新功能/复杂异常）

> 对 Demo 环境：优先把可稳定复现的只读查询做成“每日冒烟 + 回归”；写入类链路（checkout / payment / fulfill）做成“可选专项”，避免因 demo 重置/风控导致误报。

### 4.2 UI 自动化建议（cytest + Selenium）

cytest 已支持 `SELENIUM_LOG_SCREEN(driver)` 在检查点失败时截图，并通过 Vue 报告展示步骤与失败信息，适合系统测试“给业务看”的场景。

## 5. 用例组织（完全贴合 cytest 规范）

### 5.1 目录=套件（Suite），`__st__.py` 做套件级 setup/teardown

建议在 `cases/end2end/` 下按业务域建套件目录：

- `cases/end2end/__st__.py`：全局初始化（如：读取配置、创建公共会话、初始化浏览器池）
- `cases/end2end/auth/`：注册/登录/权限
- `cases/end2end/product/`：类目/搜索/详情
- `cases/end2end/cart/`：购物车
- `cases/end2end/order/`：下单/取消/支付结果
- `cases/end2end/fulfillment/`：发货/物流/收货
- `cases/end2end/aftersale/`：退款/退货/售后
- `cases/end2end/promotion/`：券/满减/秒杀（如有）
- `cases/end2end/admin/`：后台管理（如要覆盖）

每个目录下放一个 `__st__.py`，控制该业务域的“进入/退出”成本，例如：

- `suite_setup`：登录获取 token、准备测试商品
- `suite_teardown`：回收订单、清理购物车、关闭浏览器

### 5.2 标签规范（建议）

建议在 `__st__.py` 定义 tags，结合 cytest 的 `--tag/--tagnot` 过滤：

- `force_tags`：强制继承（例如 `系统测试`、`电商`）
- `default_tags`：默认标签（例如 `回归`）

用例级标签（类属性）建议包含：

- `冒烟` / `回归` / `高风险`
- `API` / `UI`
- `支付` / `库存` / `价格` 等域标签

## 6. 核心业务场景覆盖矩阵（E2E）

> 下表是“电商方方面面”的系统测试覆盖清单。你在面试中可以说：我先把系统拆成业务域，再列关键路径与风险点，然后决定哪些走 API、哪些走 UI，最后落地到自动化目录结构与标签体系。

### 6.1 用户与鉴权（Auth）

- **注册/登录/登出**
  - 正常注册（手机/邮箱）→ 登录 → 退出
  - 错误密码、锁定策略、验证码策略（如有）
- **Token 与权限**
  - Token 过期后自动刷新/强制重新登录
  - 未登录访问购物车/下单：应拦截或匿名流程
- **风控与限流**
  - 登录频次限制、短信验证码次数限制（如有）

### 6.2 商品与搜索（Product）

- **类目/搜索/筛选**
  - 搜索关键字、分页、排序（销量/价格）
  - 筛选：品牌/价格区间/规格属性
- **商品详情**
  - SKU 切换、价格/库存联动展示
  - 下架商品不可加购/不可下单
- **价格一致性（高风险）**
  - 商品详情价 vs 结算页价 vs 下单价一致

### 6.3 购物车（Cart）

- **加购**
  - 普通商品、SKU 商品加购
  - 超库存加购应提示
- **编辑**
  - 修改数量、删除、勾选/全选
- **跨端/跨会话**
  - 未登录加购 → 登录后合并购物车（如支持）

### 6.4 下单与订单（Order）

- **确认订单**
  - 选择地址、运费计算、优惠券可用性
  - 发票/备注（如支持）
- **创建订单**
  - 正常下单成功，生成订单号
  - 库存紧张下单并发（至少做接口层一致性检查）
- **取消订单**
  - 未支付取消、超时自动取消（如支持）

### 6.5 支付（Payment）

- **支付发起**
  - 生成支付单/支付二维码/支付跳转链接
- **支付回调（幂等）**
  - Demo 环境通常不会开放真实支付通道与回调；若 API 支持“模拟支付状态”，仅验证幂等与状态机
- **异常链路**
  - 订单已取消仍收到支付成功回调：系统应做一致性保护

### 6.6 履约（Fulfillment）

- **发货**
  - 管理端发货后，用户端可查询物流
- **收货**
  - 收货后订单状态变更、可评价（如支持）

### 6.7 售后（After-sale）

- **仅退款**
  - 未发货订单申请退款 → 审核通过 → 退款成功
- **退货退款**
  - 已发货申请退货 → 填写物流 → 收货后退款
- **边界条件**
  - 超过售后期不可申请
  - 部分退款（如支持）

### 6.8 营销（Promotion）

- **优惠券**
  - 领取/使用/过期/不可叠加规则
  - 满减门槛校验（含运费/不含运费）
- **秒杀/限购（如有）**
  - 限购数量、并发下单（先做接口级防超卖验证）

### 6.9 库存一致性（Inventory，横切关注点）

- 下单扣减、取消释放、退款回补（按业务规则）
- 幂等：重复扣减、重复回补的防护

### 6.10 权限与安全（Security，横切关注点）

- 越权访问：普通用户不能访问管理端 API
- 参数篡改：价格/优惠券 id/商品 id 篡改
- 关键操作审计（如支持）

## 7. 自动化落地策略（cytest 视角）

### 7.1 用例命名与粒度

- **粒度建议**：1 个用例覆盖 1 条清晰业务路径（不做“超级用例”）
- **命名建议**：`{域}-{编号}-{路径}`，例如：
  - `ORDER-0001-加购到下单成功`
  - `PAY-0003-支付回调幂等`

### 7.2 检查点策略（CHECK_POINT）

系统测试中检查点要“业务可读”：

- **关键检查点**：状态机（订单状态）、金额（应付/实付）、库存（扣减/释放）
- **辅助检查点**：字段完整性、分页总数、响应时间（阈值）

并充分利用 cytest 的检查点表达式内省能力（失败时展示左右值）。

### 7.3 执行策略（标签驱动）

建议预置 3 套执行入口：

- **冒烟**：`--tag 冒烟`
- **回归**：`--tag 回归`
- **高风险**：`--tag 高风险`

并用 `--suite` 按业务域切片（例如只跑 `order`、`payment`）。

## 9. 风险与对策

- **外部依赖不稳定（支付网关/短信）**
  - 对策：在 SIT 环境使用 Mock 或沙箱；关键链路用接口模拟支付回调
- **测试数据污染导致用例互相影响**
  - 对策：用例自建自清；必要时每天重置数据库/环境
- **UI 自动化波动**
  - 对策：UI 只覆盖关键路径；其余用 API 稳定回归；失败自动截图进报告
